# 総務人事管理システム 段階構築ロードマップ

作成日: 2026-07-07

## 位置づけ

現在のリクルート管理システムは、総務人事部が使う業務システムの第一領域として運用する。
今後は、以下の2本立てで整理する。

```text
総務人事管理システム
├─ ① リクルート管理システム
└─ ② 現職者管理システム
```

現時点で本番運用してよいのは、①リクルート管理システムの既存範囲。
②現職者管理システムは、HR秘匿情報やCore DB更新に関わるため、設計・レビュー・モックまでに留める。

## ① リクルート管理システム

現在進める領域。

主な業務:

- 学生管理
- 学生カルテ
- 面接管理
- フェア分析
- 学校分析
- 採用ROI確認
- データ品質確認
- 操作履歴確認
- 月次CSV出力
- LSTEP連携準備

当面の優先順位:

1. 総務人事部が迷わず学生カードを更新できる状態にする
2. 保存エラー、権限エラー、重複登録の再発を減らす
3. 面接管理から選考結果を更新しやすくする
4. フェア・学校分析を月次確認に使える状態にする
5. LSTEP連携再開まで、CSVとデータ品質を整える

## ② 現職者管理システム

今後設計する領域。

対象業務:

- 入社手続き
- 雇用条件確認
- 必要書類確認
- 配属予定店舗確認
- 異動手続き
- 産休・育休・休職手続き
- 復職手続き
- 退職手続き
- 社員マスタ反映待ち

扱ってはいけないもの:

- HR秘匿情報をNOV Talent側へ直接保存すること
- `employees` を画面から即時自動更新すること
- `employee_assignment_histories` をレビューなしで更新すること
- role / employee_roles をNOV Talent側から変更すること
- Secret / service_role / RLS / GRANT をレビューなしで変更すること

## リクルート管理から現職者管理へ渡す候補

候補者が内定・入社決定になった時、将来的に以下を現職者管理へ渡す。

| 項目 | 現在の正本 | 現職者管理での扱い |
| --- | --- | --- |
| 学生ID | `talent_students.id` | 入社前候補者の参照元 |
| 氏名 | `talent_students.full_name` | 社員化前の表示確認 |
| 学校 | `talent_students.school_id` または snapshot | 採用経路確認 |
| 選考結果 | `talent_students.result_status` | 入社手続き開始条件 |
| 希望店舗 | talent側の希望店舗情報 | 配属相談の材料 |
| 見学店舗履歴 | talent側の見学履歴 | 配属・相性確認 |
| 採用担当者 | `employees.id` 参照 | 引き継ぎ担当 |
| 更新者 | `employees.id` 参照 | 監査ログ |

## Core / HUBレビューが必要な項目

以下は、実装前にCore DB番人 / HUBへ戻す。

- 現職者管理用の新規 `talent_` DDL
- 入社手続きキュー
- 社員マスタ反映待ち
- employeesへの反映
- employee_assignment_historiesへの反映
- HR秘匿情報の扱い
- 通知、承認、LINE WORKS連携
- HUBカード表示や権限制御
- service_role / Secret / RLS / GRANT / RPC

レビュー依頼のたたき台:

- `Coreレビュー依頼_現職者管理_入社手続きキュー.md`

## Phase別ロードマップ

### Phase 1: リクルート管理の完成度向上

- 学生カードの項目整理
- 保存ボタンとエラー表示の改善
- 重複登録の見つけ方と管理対象外運用の整理
- 総務人事部向けマニュアル更新
- フェア・学校・面接の操作性改善

完了条件:

- 総務人事部スタッフがHUB経由で学生カードを保存できる
- 操作履歴に社員ID付きで残る
- 内定数、合格数、学生数が実データと合う
- 月次確認でCSV出力が使える

### Phase 2: 現職者管理の要件整理

- 入社手続きチェックリスト案
- 異動、休職、復職、退職のチェックリスト案
- どの情報がHR秘匿か分類
- Core DBに反映する前の承認フロー案
- Claude外部監査pack作成

完了条件:

- 総務人事部が「現職者管理で何を管理したいか」を確認できる
- Core / HUBレビューへ出せる粒度になっている

### Phase 3: UIモック

- DB変更なしで現職者管理の入口モックを作る
- 入社、異動、休職、退職のカード型導線を検証
- HR秘匿情報は入力させない
- 社員マスタ反映は「承認待ち」表示だけにする

完了条件:

- 画面の使いやすさを確認できる
- DB設計前に、項目過多や導線の問題を洗い出せる

### Phase 4: Core連携設計

- Core DB番人レビュー用DDL案
- HUB権限案
- 操作ログ案
- 承認ログ案
- 通知設計案

完了条件:

- DDL投入前のレビューが完了
- どのschema / tableに何を持つか確定
