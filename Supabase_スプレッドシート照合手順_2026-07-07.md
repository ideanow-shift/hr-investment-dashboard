# Supabase / スプレッドシート照合手順 2026-07-07

## 目的

総務人事部が運用しているスプレッドシート実データと、NOV Talent / リクルート管理システムに反映されているSupabaseデータのズレを確認する。

この手順は照合用であり、いきなりSupabaseへ一括UPDATE / DELETEしない。

## 基本方針

- 正本移行前は、総務人事部のスプレッドシートを実務上の確認元とする。
- NOV Talentの通常運用はSupabaseを参照する。
- ズレが出た場合は、まず件数、重複、管理対象外、ステータス表記揺れを確認する。
- 一括修正が必要な場合は、CSV差分を作り、レビュー後に反映する。

## 2. Supabase側をCSV出力

Supabase SQL Editorで以下を実行し、結果をCSV出力する。

### 学生一覧

```sql
select
  id,
  student_code,
  full_name,
  regexp_replace(full_name, '[[:space:]　]+', '', 'g') as normalized_name,
  school_name_snapshot,
  regexp_replace(coalesce(school_name_snapshot, ''), '[[:space:]　]+', '', 'g') as normalized_school,
  cohort,
  line_status,
  salon_tour_status,
  interview_status,
  result_status,
  offer_status,
  expected_join_status,
  management_status,
  next_action,
  next_action_date,
  updated_at
from public.talent_students
order by cohort, normalized_name, normalized_school, student_code;
```

### KPI確認

```sql
select
  count(*) filter (where management_status = '有効') as managed_students,
  count(*) filter (where management_status = '管理対象外') as inactive_students,
  count(*) filter (where management_status = '有効' and line_status = '登録済') as line_registered,
  count(*) filter (where management_status = '有効' and salon_tour_status in ('予定', '実施済')) as salon_tours,
  count(*) filter (where management_status = '有効' and interview_status in ('予定', '実施済')) as interviews,
  count(*) filter (where management_status = '有効' and result_status in ('合格', '条件付き合格', '内定', '条件付き内定')) as passed_or_offer,
  count(*) filter (where management_status = '有効' and offer_status in ('内定', '承諾')) as offered,
  count(*) filter (where management_status = '有効' and expected_join_status in ('入社予定', '入社済')) as expected_joiners
from public.talent_students;
```

### 重複候補

```sql
select
  regexp_replace(full_name, '[[:space:]　]+', '', 'g') as normalized_name,
  coalesce(school_id::text, regexp_replace(coalesce(school_name_snapshot, ''), '[[:space:]　]+', '', 'g')) as normalized_school,
  count(*) as active_count,
  string_agg(coalesce(student_code, id::text), ' / ' order by student_code) as student_codes
from public.talent_students
where management_status = '有効'
group by 1, 2
having count(*) > 1
order by active_count desc, normalized_name;
```

## 4. よくあるズレ原因

### 重複登録

同じ学生が、27卒とサロン実習の両方に入っているケース。

対応:

- 残す行を決める。
- もう一方は物理削除せず、まず `management_status = 管理対象外` にする。
- どちらを残したか操作履歴に残す。

### スペース揺れ

氏名に半角スペース、全角スペース、スペースなしが混在するケース。

対応:

- 照合時は `regexp_replace(full_name, '[[:space:]　]+', '', 'g')` で確認する。
- 表示名は人事部の正式表記に寄せる。

### ステータス表記揺れ

例:

- 合格 / 内定
- 条件付き合格 / 条件付き内定
- 入社予定 / 内定後フォロー

対応:

- 画面表示はリクルート管理向けに整理する。
- DB互換項目は残し、集計ロジックで吸収する。

### 管理対象外の扱い

CSVにはいるが、ダッシュボードKPIには含めない学生がいる。

対応:

- `management_status = 管理対象外` の件数を別で確認する。
- 実務上不要になった学生も、物理削除は避ける。

## 5. 反映前チェック

Supabaseへ修正を反映する前に、必ず以下を確認する。

- 反映対象CSVに `id` または `student_code` がある
- 物理削除ではなく管理対象外で足りるか
- 重複解消後のKPIが総務人事部の認識と合うか
- 操作履歴を残せるか
- Core DBや社員マスタに影響しないか
