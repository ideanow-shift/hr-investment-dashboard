# 総務人事管理システム 構想整理

作成日: 2026-07-06
更新日: 2026-07-07

## 目的

現在のNOV Talent / 人材投資管理は、実態としてリクルート管理システムとして機能している。
今後は、総務人事部が日常業務を一元管理できる上位システムとして整理する。

2026-07-07時点では、既存の画面・DB・GASはリクルート管理を中心に運用する。
現職者管理は、HR秘匿情報やCore DB更新に接近するため、すぐに実装せず責務境界と承認フローを固めてから段階的に進める。

上位名称案:

- 総務人事管理システム
- Human Resources Management Hub
- IDEA NOV HR Operations

## 推奨構成

```text
総務人事管理システム
├─ ① リクルート管理システム
│  ├─ 学生管理
│  ├─ フェア分析
│  ├─ 学校分析
│  ├─ 面接管理
│  └─ 内定・入社予定
│
└─ ② 現職者管理システム
   ├─ 入社手続き
   ├─ 異動手続き
   ├─ 産休・育休・休職手続き
   ├─ 復職手続き
   ├─ 退職手続き
   └─ 社員マスタ反映待ち
```

## ① リクルート管理システム

現在のNOV Talentをこの領域に位置づける。

主な対象:

- 学生
- フェア
- 学校
- 面接
- 内定
- 入社予定
- 採用ROI
- LSTEP連携

現時点の実装範囲:

- 学生フォロー
- 学生カルテ
- 面接管理
- フェア分析
- 学校分析
- データ品質確認
- 操作履歴
- 月次CSV
- LSTEP連携準備

LSTEP連携は、LINE公式アカウント側の整理が終わるまで本接続を保留する。
保留期間中は、未紐付けCSV、連携テンプレート、学生データ品質改善、総務人事部の操作性改善を優先する。

正本:

- 学生正本: `talent_students.id`
- 学校・フェア: `talent_` prefixのNOV Talent固有データ
- 社員・店舗・部署・役職・職種・権限: Core DB参照

## ② 現職者管理システム

入社決定後の総務人事部手続きを管理する領域。

主な対象:

- 入社手続き
- 雇用契約
- 必要書類
- 配属店舗
- 職種・雇用形態
- 社保・労務確認
- 異動手続き
- 休職・産休・育休・復職手続き
- 退職手続き

重要:

- HR秘匿情報を扱う可能性があるため、NOV Talentの延長で独自実装しない。
- Core DB / HUB HR Module / 認証権限の設計と合わせる。
- 社員マスタへの直接反映は、最初は自動反映ではなく「承認後反映」が安全。
- 現職者管理の画面を作る場合も、社員・店舗・部署・役職・職種はCore DBを参照する。
- 住所、家族、扶養、給与、税、保険、マイナンバーなどのHR秘匿情報は、NOV Talent側へ持ち込まない。
- 入社予定者から社員化する流れは、反映待ちキューと承認ログを前提に設計する。

## リクルート管理から現職者管理への流れ

```text
学生
↓
内定
↓
入社予定
↓
入社決定
↓
現職者管理: 入社手続き
↓
必要項目確認
↓
承認
↓
Core DB employees / assignment histories へ反映
```

## 社員マスタ反映の考え方

最初から自動で `employees` を更新するのは避ける。
Phase 1では、以下のような「反映待ちキュー」を持つ考え方が安全。

```text
手続き完了
↓
社員マスタ反映待ち
↓
総務人事部または管理者が確認
↓
承認
↓
Core DBへ反映
↓
操作履歴に記録
```

## Core DB参照方針

NOV Talent / 総務人事管理システム側では、以下を独自マスタ化しない。

| 対象 | 正本 |
| --- | --- |
| 社員 | `public.employees.id` |
| 店舗 | `public.stores.id` |
| 部署 | `public.departments.id` |
| 役職 | `public.positions.id` |
| 職種 | `public.job_types.id` |
| 権限 | `public.roles` / `public.employee_roles` |

## Phase案

### Phase 1: リクルート管理の安定化

- 学生カルテの入力項目整理
- 保存エラーの再発防止
- 重複学生の検知・管理対象外運用
- フェア、学校、面接、操作履歴の運用定着
- 総務人事部フィードバック回収

### Phase 2: 入社手続きの設計

- 内定・入社決定者から現職者管理へ引き渡す項目を定義
- 入社手続きチェックリストを作成
- Core DB / HUB HR Moduleへのレビュー依頼
- 社員マスタ反映待ちキューの設計

### Phase 3: 現職者管理の限定モック

- DB変更なしで画面モックを作成
- 入社、異動、休職、復職、退職の状態遷移を整理
- HR秘匿情報を扱わない範囲で、総務人事部に確認してもらう

### Phase 4: Core連携レビュー後の実装

- Core DB番人レビュー後にDDL案を確定
- RLS、GRANT、service_role、監査ログ方針を確定
- HUB権限、通知、承認フローを確定
- 小さく本番運用へ移行
