# スプレッドシート実データ照合・移行手順

作成日: 2026-07-07

## 目的

総務人事部が運用しているGoogleスプレッドシートの実データと、NOV Talent / Supabase上の学生データにずれが出た場合の照合手順を整理する。

この手順は、いきなり本番データを上書きするためのものではない。
まず差分を見える化し、総務人事部が確認してから反映する。

## 基本方針

- Supabaseの `talent_students` を現システムの正本として扱う
- 総務人事部スプレッドシートは実運用由来の重要な参照元として扱う
- 差分がある場合、どちらが正しいかは自動判断しない
- 物理削除はしない
- 重複や不要データは `management_status = 管理対象外` で外す
- 一括UPDATE/DELETEはCore/司令塔確認なしに実行しない

## 照合で見る項目

最低限、以下を比較する。

| 項目 | 比較目的 |
| --- | --- |
| 氏名 | 同一学生候補の特定 |
| 学校名 | 同一学生候補の精度向上 |
| 卒年 / 区分 | 27卒、28卒、サロン実習の混在防止 |
| LINE登録 | ファネル数値確認 |
| 見学状況 | 見学数のずれ確認 |
| 面接状況 | 面接数のずれ確認 |
| 選考結果 | 合格・内定数のずれ確認 |
| 管理状態 | 対象外の反映確認 |
| 流入元 | フェア・学校分析との整合 |

## 文字列正規化

照合時は、以下の揺れを吸収する。

- 半角スペース
- 全角スペース
- 前後の空白
- 学校名の軽微な表記ゆれ
- 旧字体 / 異体字は、まず候補として出し手確認する

例:

```text
伊藤 好玲
伊藤好玲
```

この2つは同一候補として扱う。

## 推奨フロー

### Step 1: スプレッドシートをCSVまたはxlsxで出力

総務人事部が運用している実データを、ローカルに保存する。

保存先例:

```text
C:\Users\bassa\Desktop\人財投資管理システム
```

### Step 2: Supabase側データをCSV出力

SQL EditorまたはダッシュボードのCSVで、現在の `talent_students` を出力する。

出力する列:

- id
- student_code
- full_name
- school_name_snapshot
- cohort
- line_status
- salon_tour_status
- interview_status
- result_status
- management_status
- source
- updated_at

### Step 3: ローカルで差分表を作る

Codex側で2ファイルを読み込み、以下の表を作る。

1. スプレッドシートにあり、Supabaseにない学生
2. Supabaseにあり、スプレッドシートにない学生
3. 両方にあるが、ステータスが違う学生
4. 氏名 + 学校名が重複している学生
5. サロン実習と27卒/28卒で二重登録されている学生

### Step 4: 差分チェック用ファイルを作成

自動反映せず、まず確認用ファイルを作成する。

推奨ファイル名:

```text
NOV_Talent_実データ照合チェック_YYYYMMDD.xlsx
```

含めるシート:

- 追加候補
- 更新候補
- 重複候補
- 管理対象外候補
- 要確認

### Step 5: 総務人事部が確認

総務人事部が、以下を判断する。

- 追加してよい
- 更新してよい
- 管理対象外にしてよい
- 保留
- 手作業確認

### Step 6: 反映

反映は以下のどちらか。

1. 少数ならダッシュボードから手作業で修正
2. 件数が多い場合は、承認済み差分だけを反映する専用SQL / GAS処理を作る

一括反映する場合は、事前にCore/司令塔へ確認する。

## 重複学生の扱い

重複候補がある場合は、以下の方針にする。

- 正しい学生レコードを1つ残す
- 残す基準は、更新日が新しい、ステータスが進んでいる、student_codeが運用上正しい、の順で確認
- 片方は削除せず `管理対象外` にする
- 理由をメモに残す

## 内定数・合格数がずれる場合

まず以下を確認する。

- `management_status = 有効` のみ集計されているか
- 同一学生が複数レコードで有効になっていないか
- サロン実習と27卒/28卒が二重に内定扱いになっていないか
- `result_status` の値が想定どおりか
- 旧項目の `offer_status` / `expected_join_status` が集計に残っていないか

## やってはいけないこと

- CSVを見て即DELETEする
- 重複候補を確認なしに一括管理対象外にする
- スプレッドシートを正としてSupabaseを丸ごと上書きする
- idなしCSVだけで更新する
- 氏名だけで同一学生と判断する
- Secretやservice_role keyをCSVや資料に含める

## Codexで支援できること

ローカルに2つのファイルがある場合、Codexで以下を支援できる。

- 列名の対応表作成
- 氏名・学校名の正規化
- 差分抽出
- 重複候補一覧作成
- 更新候補一覧作成
- 確認用xlsx作成
- 反映用SQL案作成

ただし、実データをSupabaseへ反映する前に、総務人事部確認とCore/司令塔確認を挟む。
