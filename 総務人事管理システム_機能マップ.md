# 総務人事管理システム 機能マップ

作成日: 2026-07-07

## 目的

NOV Talent / リクルート管理システムは、当初の人材投資管理から、総務人事部の日常業務を支えるシステムへ拡張しつつあります。

ただし、採用活動と入社後の労務手続きは責務が違うため、以下の2領域に分けて整理します。

```text
総務人事管理システム
├─ 1. リクルート管理システム
└─ 2. 現職者管理システム
```

## 1. リクルート管理システム

現在の実装・運用中心です。

### 扱う対象

- 求人活動
- 学校接点
- 就職フェア
- 学生接触
- LINE登録
- サロン見学
- 面接
- 選考結果
- 内定後フォロー
- 採用ROI

### 主な画面

- 全体サマリー
- フェア分析
- 学校分析
- 学生フォロー
- 面接管理
- データ品質
- 操作履歴
- 使い方

### 正本

- 学生: `public.talent_students`
- 学校: `public.talent_schools`
- フェア: `public.talent_fairs`
- 操作履歴: `public.talent_operation_logs`
- 社員: `public.employees`
- 店舗: `public.stores`
- 権限: `public.roles` / `public.employee_roles`

### 今すぐ進めてよいこと

- UI改善
- 学生カード整理
- 面接管理の既存メモ運用
- フェア/学校分析の使いやすさ改善
- 総務人事部フィードバック反映
- スプレッドシートとSupabaseの照合手順整理

## 2. 現職者管理システム

将来領域です。現時点では設計・レビュー・画面プレビューまでに留めます。

### 扱う候補

- 入社手続き
- 配属準備
- 異動手続き
- 産休・育休
- 休職
- 復職
- 退職手続き
- 社員マスタ反映待ち

### 画面候補

- 入社手続きキュー
- 手続きチェックリスト
- 異動手続き一覧
- 休職・復職手続き一覧
- 退職手続き一覧
- 承認待ち一覧
- 通知待ち一覧

### 必ずレビューへ戻すもの

- 現職者管理用 `talent_` DDL
- `employees.id` への紐付け
- `stores.id` 参照
- `employee_assignment_histories` 反映
- 承認フロー
- 通知
- HR秘匿情報の扱い
- RLS / GRANT / RPC
- service_role / Secret

### 現時点でやらないこと

- 社員マスタ更新
- 異動履歴の本番反映
- 退職ステータスの本番反映
- HR秘匿情報の保存
- 現職者管理テーブルのSQL Editor投入

## リクルート管理から現職者管理への流れ

```text
学生接触
  ↓
見学・面接
  ↓
選考結果
  ↓
内定後フォロー
  ↓
入社手続き候補
  ↓
現職者管理レビュー済みキュー
  ↓
社員マスタ反映
```

## 操作権限の考え方

### リクルート管理

- `super_admin`
- `talent_admin`
- `talent_editor`
- 必要に応じて `talent_viewer`

保存系操作は、フロントだけで確定せず、GAS / backend / Core DB側で再判定します。

### 現職者管理

現職者管理は、HR秘匿情報や社員マスタ更新に近いため、リクルート管理より厳しい権限設計が必要です。

初期案:

- 作成: 総務人事部
- 承認: 経営層または指定管理者
- 反映: backend / service_role
- 閲覧: 業務に必要な範囲のみ

## 情報の置き場所

| 情報 | 正本 | NOV Talentでの扱い |
| --- | --- | --- |
| 学生 | `talent_students` | 正本 |
| 学校 | `talent_schools` | 正本 |
| フェア | `talent_fairs` | 正本 |
| 社員 | `employees` | 参照のみ |
| 店舗 | `stores` | 参照のみ |
| 部署 | `departments` | 参照のみ |
| 役職 | `positions` | 参照のみ |
| 職種 | `job_types` | 参照のみ |
| 権限 | `roles` / `employee_roles` | 参照のみ |
| HR秘匿情報 | HR Module | 持ち込まない |

## 直近の進め方

1. リクルート管理の実操作フィードバックを反映する。
2. 学生カード、フェア分析、学校分析、面接管理を使いやすくする。
3. 現職者管理は画面プレビューと資料で認識合わせを進める。
4. DB化が必要になった時点で、Core DB番人 / HUBレビューへ戻す。
5. 承認後に、入社手続きキューから段階実装する。

## 判断

現時点では、リクルート管理を運用しながら、現職者管理を設計プレビューとして育てるのが安全です。

総務人事部の業務範囲は広がりますが、社員マスタやHR秘匿情報に近い処理は、必ずCore DB / HUB / HR Moduleの境界を確認してから進めます。
