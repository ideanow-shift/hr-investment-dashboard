# NOV Talent Supabase移行設計書

## 目的

NOV Talentのデータ管理をGoogleスプレッドシート + GASからSupabaseへ移行し、本格的な人財投資管理システムとして運用できる状態にする。

目的は以下です。

- 学生データをDBで安全に管理する
- ダッシュボードからの追加・更新を安定させる
- 操作履歴、更新者、更新日時をDBで保持する
- Lステップ連携に備える
- 将来的な認証、権限、分析拡張に耐える構造にする

## 移行方針

スプレッドシート運用は停止予定のため、Supabaseを正規DBとする。

ただし、移行直後はデータ確認のため、スプレッドシートはバックアップとして残す。

## DB構成

### app_settings

年度ごとの目標値を管理する。

- 年度
- 採用目標人数
- 接触人数目標
- 面接成約目標
- 採用予算
- 入社予定数

### schools

学校マスタ。

学校名の表記ゆれ対策として、今後はこのテーブルを正にする。

### fairs

フェア実績。

費用、接触数、LINE登録数、見学取得数を保持する。

### students

学生個別データ。

27卒、28卒、サロン実習は `cohort` で区分する。

管理対象外は削除せず `management_status` で管理する。

### operation_logs

学生追加、更新、管理対象外変更などの操作履歴。

### student_followups

将来的なフォロータスク管理用。

今は学生テーブルの `next_action` と `next_action_date` でも運用できるが、タスク履歴を細かく追う場合はこちらへ分離する。

## 画面側の変更方針

現在:

```text
GitHub Pages → GAS JSONP → Googleスプレッドシート
```

移行後:

```text
GitHub Pages → Supabase API → PostgreSQL
```

変更対象:

- `app.js` のデータ取得処理
- 学生追加処理
- 学生更新処理
- 操作履歴保存処理
- データ更新ボタン

## 認証方針

本番運用ではSupabase Authを使う。

初期候補:

- 総務人事部メンバーのみログイン可能
- メールログイン
- 店舗管理者は閲覧のみ
- 総務人事部は追加・更新可能

## RLS方針

Supabaseをフロントエンドから直接使う場合、RLSは必須。

初期検証では `authenticated` ユーザーに読み書きを許可する。

本番では以下に分ける。

- `hr_admin`: 全件読み書き
- `manager`: 閲覧のみ
- `executive`: 集計閲覧のみ

## Lステップ連携の拡張

今後、以下のテーブル追加を検討する。

- `line_accounts`
- `line_messages`
- `line_events`

学生とLINE IDを紐づけることで、以下が可能になる。

- LINE登録状況の自動反映
- 見学リマインド
- 面接リマインド
- 内定後フォロー
- 返信状況の可視化

## 移行完了条件

- Supabaseに全学生データが入っている
- ダッシュボードのKPIが現行版と一致している
- 学生追加・更新がSupabaseに保存される
- 操作履歴がSupabaseに残る
- 管理対象外運用が機能する
- 総務人事部がダッシュボード操作で日常運用できる
