# 総務人事部フィードバック回収表

## 目的

総務人事部に実際に操作してもらい、リクルート管理システムと将来の現職者管理システムを完成に近づけるためのフィードバック回収表です。

感想だけでは修正優先順位を決めづらいため、必ず「どの画面で」「何をしようとして」「どう困ったか」を残します。

## 回収ルール

- 保存できない、画面が固まる、数字が合わないものは最優先で記録する。
- 使いづらい、項目が多い、どこを押せばよいかわからないものはUI改善として記録する。
- 追加したい項目は、すぐDB追加せず「既存項目で代用できるか」を先に確認する。
- 個人情報や学生の詳細メモをそのまま外部監査資料へ貼らない。

## 分類

| 分類 | 例 | 優先度 |
| --- | --- | --- |
| 保存エラー | 学生カードが保存できない、権限なしになる | 最優先 |
| データ不整合 | 内定数、接触数、学校別人数がスプレッドシートと違う | 高 |
| 学生カード編集 | 項目が多い、入力順がわかりづらい | 高 |
| 面接管理 | 診断メモ、最終判断、複数名面接の扱い | 中 |
| フェア/学校 | フェア追加、学校詳細、学校訪問メモ | 中 |
| 現職者管理 | 入社手続き、退職、異動、休職の導線 | 中 |
| 権限/ログイン | HUBから入れない、社員IDが記録されない | 高 |
| スマホ操作 | 遅い、開かない、ボタンが押しづらい | 高 |
| 表記/文言 | NOV Talent表記、リクルート管理表記の違い | 低 |

## 総務人事部に依頼する確認順

1. HUBからリクルート管理システムを開けるか。
2. 右上にHUB社員情報が表示されるか。
3. 学生フォローで学生カードを開けるか。
4. 次アクションとメモを更新して保存できるか。
5. 操作履歴に自分の名前・社員ID付きで残るか。
6. 面接管理で対象学生を探せるか。
7. 現職者管理タブで入社手続き候補の意味がわかるか。
8. スマホから開いた時に待てる速度か。

## 対応優先順位

### すぐ直す

- 保存できない
- 権限があるはずなのに権限なしになる
- 数字が明らかに違う
- 学生カードが開かない
- スマホで開けない

### 次に直す

- 項目が多い
- 入力順がわかりづらい
- ボタンの場所がわかりづらい
- 検索しても見つけにくい
- 面接管理の判断メモが使いづらい

### Coreレビュー後に進める

- 入社手続きキューのDB化
- 現職者管理の手続き履歴テーブル
- 社員マスタへの反映
- 異動、退職、休職、復職の正本管理
- 通知、承認、LSTEP本接続

## 対応時に止めること

- 学生データの物理削除
- Supabaseの一括UPDATE/DELETE
- Core DBのemployees/stores/roles直接変更
- role / employee_rolesの変更
- Secretやservice_role keyの変更
- LSTEP本接続

## 使い方

総務人事部から口頭・チャット・Google Docsで来た内容を、この表の分類に合わせて転記します。

転記後、以下の順で判断します。

1. 既存UIだけで直せるか
2. GAS変更が必要か
3. Supabase既存データの確認が必要か
4. Core DB番人レビューが必要か
5. HUB/Core/HR Moduleへ相談が必要か

迷った場合は、DBや権限を触らず、まず再現条件だけを整理します。
