# 総務人事部向け 現職者管理ヒアリングシート

作成日: 2026-07-07

## 目的

リクルート管理システムで内定・入社決定となった学生を、将来の現職者管理システムへ安全に引き継ぐため、総務人事部の日常業務を整理する。

現時点では設計確認用であり、Core DB、社員マスタ、HR秘匿情報、権限設定は変更しない。

## 前提

- リクルート管理システムは、学生・面接・フェア・学校・採用ROIを扱う。
- 現職者管理システムは、入社後または入社直前の社内手続きを扱う。
- 社員情報の正本はCore DBの `employees`。
- 店舗情報の正本はCore DBの `stores`。
- HR秘匿情報はNOV Talent側へ直接保存しない。
- 社員マスタへの反映は、将来的にも承認フローを通す。

## 1. 入社手続き

確認したいこと:

- 入社決定後、総務人事部が最初に確認する項目は何か。
- 入社手続きで必ず完了確認したい書類は何か。
- 入社日前に完了していないと困る項目は何か。
- 店舗配属の確定前に、誰の承認が必要か。
- 入社予定者への連絡履歴を残す必要があるか。

候補項目:

- 入社予定日
- 配属予定店舗
- 雇用形態
- 必要書類回収
- 雇用契約確認
- 制服・備品確認
- 研修初日案内
- 店舗への共有
- 社員マスタ登録待ち

## 2. 異動手続き

確認したいこと:

- 異動依頼は誰が起票するか。
- 異動元店舗、異動先店舗、異動日以外に必要な項目は何か。
- 異動前後で通知すべき相手は誰か。
- 異動履歴として残すべき理由分類はあるか。

候補項目:

- 対象社員
- 異動元店舗
- 異動先店舗
- 異動予定日
- 異動理由
- 承認者
- 店舗共有状況
- 社員マスタ反映待ち

## 3. 産休・育休・休職・復職

確認したいこと:

- 産休、育休、休職、復職で管理項目は分けるべきか。
- 休職種別はCore DBのどの項目と連動する想定か。
- 復職予定日や復職面談の管理が必要か。
- 店舗側へ共有する範囲と、秘匿すべき範囲は何か。

候補項目:

- 対象社員
- 手続き種別
- 開始日
- 終了予定日
- 復職予定日
- 必要書類
- 本人連絡状況
- 店舗共有状況
- HR秘匿メモの扱い

## 4. 退職手続き

確認したいこと:

- 退職手続きで必ず追う期限は何か。
- 退職理由はどの粒度で持つべきか。
- 貸与物返却やアカウント停止を管理する必要があるか。
- 社員マスタの就労ステータス変更は誰が承認するか。

候補項目:

- 対象社員
- 退職予定日
- 最終出勤日
- 退職理由分類
- 必要書類
- 貸与物返却
- アカウント停止確認
- 店舗共有状況
- 社員マスタ反映待ち

## 5. 権限と操作

確認したいこと:

- 総務人事部の誰が全操作できるべきか。
- 店長、教育担当、本部スタッフは閲覧だけでよいか。
- 社員マスタ反映前の承認者は誰か。
- 操作履歴に必ず残したい項目は何か。

候補:

- 総務人事部: 作成・更新・確認
- 経営層: 閲覧・承認
- 店長: 自店舗関連の閲覧
- 教育担当: 研修対象者の閲覧

## 6. Core / HUBレビューが必要になるもの

以下は実装前にCore DB番人 / HUBへ戻す。

- 現職者管理用の新規 `talent_` テーブル
- 社員 `employees.id` への紐付け
- 店舗 `stores.id` への紐付け
- 社員マスタ反映待ちキュー
- 承認フロー
- 通知
- 操作ログ
- 権限制御
- HR秘匿情報の扱い

## 7. まず決めたいこと

1. 現職者管理で最初に作るべき画面は「入社手続き」でよいか。
2. 入社手続きは、学生から社員になる前のキューとして扱うか。
3. 社員マスタへの反映は、必ず手動承認にするか。
4. 産休・育休・休職・復職・退職は、初期はチェックリスト管理でよいか。
5. HR秘匿情報をこのシステムに入れない運用で問題ないか。
