# Claude外部監査 現職者管理構想pack

作成日: 2026-07-07

## 対象

リクルート管理システム / NOV Talent から発展する、総務人事部向けの現職者管理構想。

## 背景

現在のシステムは、学生、フェア、学校、面接、内定、採用ROIを扱うリクルート管理システムとして運用しています。

今後、総務人事部の業務をさらに一元化するため、内定・入社決定後の手続きを扱う `現職者管理システム` の構想を検討しています。

想定する上位構成:

```text
総務人事管理システム
├─ ① リクルート管理システム
└─ ② 現職者管理システム
```

## Claudeへ確認してほしいこと

1. リクルート管理と現職者管理の責務分離は妥当か
2. 内定・入社決定者を現職者管理へ引き継ぐ導線は自然か
3. 入社、異動、休職、復職、退職の画面構成が総務人事部の日常運用に合いそうか
4. HR秘匿情報をNOV Talent側へ持ち込まない方針は十分に明記されているか
5. 社員マスタ反映を直接更新ではなく、承認待ちキューにする考え方は安全か
6. Core DB / HUB / HR Moduleレビューへ戻すべき境界が明確か
7. 総務人事部へヒアリングする質問が不足していないか
8. 入社手続き候補の不足サマリーと確認ラベルは、総務人事部の初期運用に役立つか

## 監査対象外

以下は監査資料に含めません。

- 実学生データ
- 実社員データ一覧
- Supabase service_role key
- GAS Script Properties
- LSTEP APIキー
- LINE WORKS Secret
- マイナンバー、給与、税、保険、銀行、住所詳細、家族/扶養情報

## 現時点で実装していないこと

- 現職者管理の本番画面
- 現職者管理用DDL
- `employees` へのINSERT / UPDATE
- `employee_assignment_histories` へのINSERT / UPDATE
- HR秘匿情報の保存
- 承認フローの本番実装
- 通知連携
- Edge Function deploy

## 現時点の方針

現職者管理は、すぐに本番実装しません。

まずは、以下を固めます。

- 業務フロー
- 画面構成
- 情報分類
- HR秘匿情報の境界
- Core DB / HUB / HR Moduleの責務分担
- 承認待ちキューの必要性

## Claudeへの依頼文

```text
以下の現職者管理構想について、総務人事部の日常運用・情報管理・システム責務分離の観点から外部監査をお願いします。

現在はリクルート管理システムが運用中心で、現職者管理は設計段階です。
DB変更、社員マスタ更新、HR秘匿情報保存はまだ行いません。

特に、リクルート管理から現職者管理への引継ぎ、社員マスタ反映待ちキュー、入社/異動/休職/退職の画面構成、Core DB / HUB / HR Moduleへ戻すべき境界について確認してください。

画面上では、入社手続き候補に不足サマリーと確認ラベルを出し、社員マスタ反映前に配属希望店舗、見学店舗履歴、次アクション日を確認できる形にしています。この導線が総務人事部の運用に合うかも見てください。

実データ、Secret、service_role key、HR秘匿情報は添付しません。
```
