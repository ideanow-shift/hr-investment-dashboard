# Claude外部監査pack: 総務人事管理システム拡張構想

作成日: 2026-07-07

## 監査してほしい対象

NOV Talent / リクルート管理システムを、総務人事部が使う上位システムとして拡張する構想です。

現在の実装は、採用・求人・学生フォロー・面接・フェア/学校分析を扱う `リクルート管理システム` です。
将来的に、入社手続きや退職手続きなどを扱う `現職者管理システム` を追加する案があります。

## 現在の整理

```text
総務人事管理システム
├─ ① リクルート管理システム
└─ ② 現職者管理システム
```

### ① リクルート管理システム

実装済み・運用中の範囲。

- 学生管理
- 学生カルテ
- 面接管理
- フェア分析
- 学校分析
- 採用ROI
- データ品質
- 操作履歴
- 月次CSV
- LSTEP連携準備

### ② 現職者管理システム

現時点では構想・設計段階。

- 入社手続き
- 配属確認
- 異動手続き
- 休職・産休・育休・復職手続き
- 退職手続き
- 社員マスタ反映待ち

## 重要な制約

以下は実行していません。

- DDL投入
- Core DB更新
- `employees` 更新
- `employee_assignment_histories` 更新
- role / employee_roles変更
- GRANT / RLS / RPC変更
- Secret / service_role変更
- HR秘匿情報の保存
- 現職者管理の本番実装

## Core DB / HUB方針

- 社員正本: `public.employees.id`
- 店舗正本: `public.stores.id`
- 部署正本: `public.departments.id`
- 役職正本: `public.positions.id`
- 職種正本: `public.job_types.id`
- 権限正本: `public.roles` / `public.employee_roles`
- 学生正本: `public.talent_students.id`
- Talent固有データは `talent_` prefix
- service_role keyはフロントに出さない
- Phase初期はGAS backend + service_role経由

## 監査観点

以下を見てください。

1. リクルート管理と現職者管理を同じ上位システムに置く設計は妥当か
2. 現職者管理をNOV Talent配下に置くべきか、HUB HR Module側に寄せるべきか
3. 入社決定者を社員マスタに反映する前の中間キューは必要か
4. 入社・異動・休職・退職の承認フローはどこで持つべきか
5. HR秘匿情報を避けるために、画面・DB・CSVで注意すべき点は何か
6. 総務人事部が日常運用する上で、操作項目が増えすぎない分け方は何か
7. Core DB正本とTalent固有データの境界に問題はないか
8. LSTEP連携を2か月程度保留する間に進めるべき安全な作業は何か
9. 入社手続き候補の不足サマリーを、社員マスタ反映前の確認導線として使う設計は妥当か
