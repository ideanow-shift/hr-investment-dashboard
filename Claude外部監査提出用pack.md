## 目的

脇田さんがClaudeへ直接提出するための外部監査packです。

監査では、直近の保存エラー修正、学校詳細ボタン修正、学生カードUI整理、今後のフェア/学校訪問管理拡張方針について、実装リスクと運用上の分かりやすさを確認してください。

## Claudeへの依頼文

以下のNOV Talent / リクルート管理システムについて、公開・運用前の外部監査をお願いします。

確認してほしい観点は以下です。

- 総務人事部が日常操作しやすいUIになっているか
- 学生カードの保存導線、ステータス変更、入力項目整理に不自然な点がないか
- 保存エラー修正が過剰な制約解除になっていないか
- 学校分析の詳細ボタン修正に副作用がなさそうか
- GAS / 既存データ / Google Sheets / Supabase Core DB への影響説明が妥当か
- Secret、service_role key、個人情報が資料やフロントに露出していないか
- 今後のフェア追加項目、学校訪問管理、面接管理をDB化する前にCore DBレビューへ戻すべき点が整理されているか

なお、今回の監査資料には実学生データの一覧、API Secret、Supabase service_role key、LSTEP APIキーは含めません。

## 直近の対応内容

### 1. 保存エラー修正

対象:

- 学生カードのステータス変更保存
- 例: 内定者のステータス変更時に保存エラーになるケース

対応:

- フロント側の過剰なステータス矛盾チェックを緩和
- GAS側の過剰なステータス矛盾チェックを緩和
- 氏名、学校、重複、次アクション日の基本検証は維持

監査ポイント:

- 緩和したことで、明らかな不正データが入りやすくなっていないか
- 総務人事部の実運用に対して、保存できないストレスを減らせているか

### 2. 学校詳細ボタン修正

対象:

- 学校分析タブの「詳細」ボタン

対応:

- 学校名キー検索ではなく、描画済み配列のindex参照へ変更
- 表記ゆれや学校名の微妙な差で詳細が開かない問題を避ける

監査ポイント:

- 並び替え後も正しい学校詳細が開くか
- 表示データと詳細データがずれないか

### 3. 学生カードUI整理

対象:

- 学生カルテ / 基本・選考タブ

対応:

- 「内定・入社ステータス」の表示項目を削除
- 既存データを壊さないため、保存時のhidden値は保持
- 選考結果を中心に入力する形へ寄せた

監査ポイント:

- 入力項目を減らしたことで、総務人事部が迷いにくくなっているか
- hidden値保持により既存データを意図せず消さない設計になっているか

### 4. フェア/学校訪問管理の拡張設計

対象:

- フェア追加項目
- 学校訪問管理ページ

追加希望:

- フェア: 担当者、備考、会場名、運営会社
- 学校訪問: 学校名、訪問日、訪問者、担当先生氏名、持参物、会話内容

現状:

- 実装やSQL投入は未実行
- Core DBレビュー依頼メモのみ作成済み

監査ポイント:

- 既存データ構造へ無理に追加せず、Core DBレビューに戻す判断が妥当か
- DDL前にUI仮設計として進められる範囲があるか

## 影響範囲

### GAS / API

- `gas/Code.gs` の学生ステータス検証ロジックに変更あり
- Apps Scriptへ貼り替え、保存、新しいデプロイが必要
- SecretやScript Propertiesの変更はなし

### Google Sheets

- 今回の修正では、Google Sheetsの列追加や既存シート構造変更はなし
- フェア項目追加、学校訪問管理を実装する場合は、既存データ項目の追加検討が必要

### Supabase / Core DB

- 今回の修正では、DDL、UPDATE、DELETE、GRANT、RLS policy変更はなし
- 新規 `talent_` テーブル追加は未実行
- 将来、面接履歴、学校訪問、フェア拡張をDB正本化する場合はCore DB番人レビューが必要

### GitHub Pages / フロント

- `app.js` のUI/保存前制御に変更あり
- `style.css` は今回の監査対象に含めるが、直近の保存エラー修正の主対象ではない
- Secretやservice_role keyはフロントへ出さない方針を維持
