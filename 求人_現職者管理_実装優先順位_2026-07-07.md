# 求人管理 / 現職者管理 実装優先順位 2026-07-07

## 前提

人財投資管理は、現在のリクルート管理システムから、総務人事部が日常運用する総務人事管理システムへ広げる。

```text
総務人事管理システム
├─ リクルート管理システム
│  ├─ 求人投資
│  ├─ フェア分析
│  ├─ 学校分析
│  ├─ 学生フォロー
│  └─ 面接管理
└─ 現職者管理システム
   ├─ 入社手続き
   ├─ 配属・異動手続き
   ├─ 休職・復職手続き
   └─ 退職手続き
```

現時点では、現職者管理は設計・画面プレビューまでに留める。社員マスタ、HR秘匿情報、承認フロー、通知、Core DB更新はCore / HUBレビュー後に進める。

## 優先順位

### Phase 1: リクルート管理の実運用安定

目的:

- 総務人事部が学生データをダッシュボード上で更新できる。
- フェア、学校、学生、面接の状態が実データと一致する。
- 保存エラー、重複、ステータス矛盾を減らす。

進めること:

- 学生カードUIの簡略化を継続。
- 保存エラーや権限エラーの再発を記録し、原因を分類する。
- Supabaseと総務人事部スプレッドシートの差分照合を定期的に行う。
- 面接管理は、既存のNotebookLM診断運用を整理して記録欄を使いやすくする。
- LSTEP本接続は保留し、未紐付けCSVと設計資料を整える。

完了条件:

- 総務人事部の主要メンバーが学生追加・更新・検索・CSV出力をできる。
- 内定数、合格数、接触数などの主要KPIが実データと一致する。
- 操作履歴にHUB社員情報が残る。

### Phase 2: 入社手続き候補の見える化

目的:

- リクルート管理で内定・承諾・入社予定になった学生を、現職者管理の入口として確認できる。
- ただし社員マスタへはまだ反映しない。

進めること:

- 現職者管理タブの入社手続き候補を確認しやすくする。
- 入社手続きに必要な確認項目を総務人事部から回収する。
- 社員化前に必要な情報と、社員化後にCore DBで持つ情報を分ける。
- Coreレビュー用の入社手続きキューDDL案を更新する。

完了条件:

- 内定後フォロー対象から入社手続き候補を一覧で確認できる。
- どの情報を学生側に残し、どの情報を社員マスタへ渡すか整理できている。

### Phase 3: 現職者管理MVP

目的:

- 入社、異動、休職・復職、退職の手続きをチェックリストとして管理する。
- 直接社員マスタを更新せず、反映待ちキューとして安全に扱う。

進めること:

- 入社手続きキューをCoreレビューへ出す。
- 承認フローと操作履歴の記録方針を決める。
- 社員、店舗、部署、役職、職種はCore DB正本を参照する。
- HR秘匿情報はHUB HR Module側に寄せ、NOV Talentには持たせない。

完了条件:

- 入社手続きの進捗がダッシュボードで確認できる。
- 社員マスタ反映前に承認・確認が入る。
- Core DB番人レビュー済みの範囲だけ実装されている。
