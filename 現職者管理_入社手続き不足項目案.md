# 現職者管理 入社手続き不足項目案

## 目的

リクルート管理で内定・承諾・入社予定になった学生を、現職者管理へ引き継ぐ前に、どの情報が不足しているか確認できるようにする。

現時点ではDB追加や社員マスタ反映は行わず、画面表示・ヒアリング・Coreレビュー用の整理に留める。

## 入社手続き候補に表示したい情報

### リクルート管理から引き継ぐ情報

- 氏名
- 学校名
- 卒年
- 流入元
- 選考結果
- 内定後フォロー状況
- 入社予定ステータス
- 配属希望店舗
- 見学店舗履歴
- 次アクション
- 次アクション日
- 担当者
- メモ

### 入社手続き前に確認したい情報

- 入社予定日
- 配属予定店舗
- 勤務希望エリア
- 雇用形態
- 職種
- 必要書類の回収状況
- 内定承諾書の回収状況
- 入社前面談の実施状況
- 制服・備品・アカウント準備状況
- 店舗共有状況

## 表示上の不足判定案

画面では、以下のような不足ラベルを出すと総務人事部が確認しやすい。

```text
不足: 入社予定日
不足: 配属予定店舗
不足: 必要書類
不足: 店舗共有
確認: 内定後フォロー
```

## 画面案

現職者管理タブの `入社手続き候補` に、以下を表示する。

- 学生名
- 学校名 / 卒年
- 選考結果
- 入社予定ステータス
- 配属希望店舗
- 不足ラベル
- 学生カードを開くボタン

最初はリクルート管理側の学生カードへ遷移するだけにする。
社員マスタ登録、社員番号採番、店舗配属反映は行わない。

## Coreレビューが必要になるタイミング

以下を行う場合はCore DB番人レビューへ戻す。

- 入社手続きキューをDB化する。
- `employees` へ新入社員を作成する。
- `stores.id` を配属予定店舗として保存する。
- `job_types.id` を職種として保存する。
- `employee_assignment_histories` へ配属履歴を作る。
- HR秘匿情報や書類ステータスを扱う。
- 承認フローや通知を作る。

## まだ行わないこと

- DDL投入
- Supabase本番データ更新
- 社員マスタ更新
- HR秘匿情報の保存
- Secret / service_role変更
- Edge deploy

## 次のUI改善候補

1. `入社手続き候補` に不足ラベルを表示する。
2. `学生カードを開く` ボタンを残し、編集は既存学生カードで行う。
3. 不足ラベルの集計を現職者管理タブ上部に表示する。
4. Coreレビュー前は、社員化ボタンや社員マスタ反映ボタンを表示しない。
