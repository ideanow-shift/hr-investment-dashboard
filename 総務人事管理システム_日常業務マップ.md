# 総務人事管理システム 日常業務マップ

## 目的

総務人事部が、採用前から入社後の社内手続きまでを同じ流れで確認できるようにするための業務マップです。

現時点の実装中心は `リクルート管理システム` です。
`現職者管理システム` は、Core DB / HUB / HR Moduleの境界を確認しながら段階的に進めます。

## 全体像

```text
総務人事管理システム
├─ リクルート管理システム
│  ├─ 学生フォロー
│  ├─ 面接管理
│  ├─ フェア分析
│  ├─ 学校分析
│  └─ 採用ROI / 目標管理
└─ 現職者管理システム
   ├─ 入社手続き
   ├─ 配属・異動手続き
   ├─ 産休・育休・休職・復職手続き
   └─ 退職手続き
```

## 日常業務の入口

| タイミング | 使う領域 | やること | 現在の状態 |
| --- | --- | --- | --- |
| フェア・学校接点後 | リクルート管理 | 学生追加、LINE登録、見学予定、次アクション登録 | 実装済み |
| 面接前後 | リクルート管理 | 面接対象確認、NotebookLM診断メモ、最終判断記録 | 既存メモ運用で実装済み |
| 内定・承諾後 | リクルート管理 | 内定後フォロー、入社手続き候補として確認 | 実装済み |
| 入社準備開始 | 現職者管理 | 入社手続きキューへ引き継ぎ | 設計・レビュー待ち |
| 入社後 | 現職者管理 | 配属、異動、休職、復職、退職手続き管理 | 設計段階 |

## リクルート管理で完結すること

- 学生情報の追加・更新
- LINE登録、見学、応募、面接、選考結果の更新
- 面接メモ、NotebookLM診断要約、面接実施者所感の記録
- フェア別ROI確認
- 学校別投資効果確認
- 採用目標、接触目標、面接成約目標、予算の確認
- 操作履歴の確認

## 現職者管理へ渡す候補

リクルート管理で以下の状態になった学生は、将来の現職者管理へ引き継ぐ候補です。

| 条件 | 引き継ぎ先 | 引き継ぎ理由 |
| --- | --- | --- |
| 選考結果が `内定` | 入社手続き候補 | 入社意思確認、書類準備、配属希望確認が必要 |
| 選考結果が `条件付き内定` | 入社手続き候補 / 要確認 | 条件確認後に入社準備へ進める |
| 入社意思あり | 入社手続き | 社員マスタ作成前の確認キューが必要 |
| 配属希望店舗あり | 配属確認 | Core DBのstores.id参照へ寄せる必要 |
| 見学店舗履歴あり | 配属判断補助 | 店舗相性や教育体制の判断材料 |

## 現職者管理で扱うが、まだ本番実装しないこと

以下はCore DB、HUB、HR Moduleに影響するため、レビューなしでは実装しません。

- employeesへの新規社員登録
- stores、departments、positions、job_typesの直接更新
- employee_assignment_historiesへの異動履歴登録
- 休職、復職、退職の正本管理
- HR秘匿情報の保存
- 給与、保険、税、マイナンバー、扶養、銀行口座情報の保存
- role / employee_rolesの変更
- 通知、LINE WORKS、LSTEPの本接続

## 総務人事部の日常操作を軽くする方針

### 毎日触る項目

- 学生フォロー
- 次アクション
- 見学予定、面接予定、選考結果
- 操作履歴

### 必要時だけ触る項目

- フェア追加
- 学校訪問メモ
- 面接診断メモ
- 目標編集
- データ品質確認

### 管理者だけ触る項目

- スプレッドシート直接編集
- GASデプロイ
- Supabase照合
- Core DBレビュー依頼
- 将来の現職者管理DB設計

## 判断

今すぐ完成形の労務管理システムを作るのではなく、まずは `リクルート管理で確定した人を、現職者管理の入口へ安全に渡す` ところまでをMVPにします。

これにより、総務人事部は現在の採用管理を止めずに、段階的に入社後手続きまで同じシステムで扱えるようになります。
