# 現職者管理 画面設計案

更新日: 2026-07-07

## 位置づけ

現職者管理は、リクルート管理システムの次工程として検討する総務人事部向け領域です。

ただし、労務・社員マスタ・HR秘匿情報に接近するため、現時点では画面設計案までに留めます。DB化、Core DB更新、HR情報保存はCore DB / HUB / HR Moduleレビュー後に進めます。

## 画面構成案

```text
総務人事管理システム
├─ リクルート管理
│  ├─ 全体サマリー
│  ├─ フェア分析
│  ├─ 学校分析
│  ├─ 学生フォロー
│  ├─ 面接管理
│  └─ データ品質
└─ 現職者管理
   ├─ 入社手続き
   ├─ 異動手続き
   ├─ 休職・復職手続き
   ├─ 退職手続き
   ├─ 社員マスタ反映待ち
   └─ 操作・承認履歴
```

## 1. 入社手続き

### 目的

内定・承諾済みの学生を、社員化する前のチェック対象として管理します。

### 主な表示

- 氏名
- 学校名
- 入社予定日
- 配属予定店舗
- 職種
- 雇用形態
- 必要書類ステータス
- 面談・説明完了状況
- 社員マスタ反映可否

### 操作

- 入社手続きチェック
- 不足書類メモ
- 配属予定店舗確認
- 社員マスタ反映申請

### 注意

社員マスタへ直接反映せず、承認待ちキューを経由します。

## 2. 異動手続き

### 目的

店舗異動、部署異動、役職変更などの手続きを管理します。

### 主な表示

- 対象社員
- 現所属店舗
- 新所属店舗
- 現部署
- 新部署
- 現役職
- 新役職
- 異動予定日
- 承認状況

### 注意

社員・店舗・部署・役職の正本はCore DBです。NOV Talent側で独自マスタを作りません。

## 3. 休職・復職手続き

### 目的

産休、育休、休職、復職の手続き進捗を管理します。

### 主な表示

- 対象社員
- 休職種別
- 開始日
- 復職予定日
- 必要書類
- 担当者
- 次アクション

### 注意

休職理由や健康情報などの秘匿性が高い情報は、NOV Talent側へ保存しません。HR Module側の設計対象です。

## 4. 退職手続き

### 目的

退職に伴う確認事項、書類、貸与物、最終出勤などを管理します。

### 主な表示

- 対象社員
- 退職予定日
- 最終出勤日
- 貸与物返却
- 書類対応
- 引き継ぎ状況
- 社員マスタ反映状況

### 注意

退職理由などの秘匿情報は、必要最小限の分類に留めます。

## 5. 社員マスタ反映待ち

### 目的

入社、異動、休職、復職、退職などの手続きを、Core DBへ反映する前の承認キューとして管理します。

### 主な表示

- 申請種別
- 対象社員または入社予定者
- 変更前
- 変更後
- 申請者
- 承認者
- ステータス
- 反映予定日

### 操作

- 申請内容確認
- 差し戻し
- 承認
- 反映済みにする

### 注意

この画面はCore DB更新に接近するため、実装前にDDL/RLS/RPC/権限設計レビューが必要です。

## 6. 操作・承認履歴

### 目的

誰が、いつ、何を申請・承認・更新したかを記録します。

### 最低限の記録

- actor_employee_id
- action
- target_type
- target_id
- result
- reason
- occurred_at

## 初期UI方針

- 総務人事部の日常操作は「今日やること」から入れる
- 入社、異動、休職、退職を大きなカードで分ける
- 社員マスタ反映は一段深い管理者画面に置く
- 申請・承認・反映を1画面で混ぜない
- 秘匿情報は表示しすぎない

## 実装前レビューが必要なもの

- 新規 `talent_` または `hr_` テーブルの要否
- Core DBの `employees` 更新フロー
- `employee_assignment_histories` との接続
- HR Moduleとの責務分担
- 権限ロール
- RLS / GRANT / RPC / Edge Function
