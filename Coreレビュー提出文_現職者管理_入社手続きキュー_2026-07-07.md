# Coreレビュー提出文: 現職者管理 / 入社手続きキュー

作成日: 2026-07-07

## 相談先判定

★Core DB番人 v2

## 理由

NOV Talent / リクルート管理システムで内定・承諾・入社予定になった学生を、将来の現職者管理へ引き継ぐための `入社手続きキュー` を検討しています。

学生正本は `public.talent_students.id`、社員正本は `public.employees.id`、店舗・部署・役職・職種はCore DBマスタを参照する方針です。

## 現在の状態

- 画面上には `現職者管理` タブを追加済み
- 入社手続き候補、準備確認済み、配属希望未確認、見学履歴要確認、次アクション日未設定を表示済み
- 現時点ではDB変更なしの設計プレビュー
- SQL Editor投入、DDL、RLS、GRANT、RPC、社員マスタ更新は未実行

## レビューしてほしいこと

`talent_` 領域として、以下2テーブル案を進めてよいか確認したいです。

- `public.talent_employee_onboarding_cases`
- `public.talent_employee_onboarding_check_items`

主な設計方針:

- `student_id` は `public.talent_students(id)` を参照
- 作成者、更新者、承認者、完了者は `public.employees(id)` を参照
- 配属予定店舗は `public.stores(id)` を参照
- 部署、役職、職種が必要な場合は `departments / positions / job_types` を参照
- HR秘匿情報は持たない
- 物理削除ではなく `is_active=false` / `case_status='cancelled'`
- Phase 1はGAS backend + service_role経由
- `anon` / `authenticated` 直接書き込みpolicyなし
- `delete` grantなし

## 確認したい判断

1. 入社手続きキューはNOV Talent側の `talent_` テーブルとして作ってよいか
2. それともHR Module側へ切り出すべきか
3. `converted_employee_id` として社員化後の `employees.id` をTalent側に保持してよいか
4. 承認者は `talent_admin` でよいか、HR系権限も必要か
5. 社員化処理はNOV Talentから起動してよいか、HR/Core側の専用フローへ渡すべきか
