# 現職者管理 MVP優先順位

作成日: 2026-07-07

## 前提

現職者管理は、リクルート管理システムから発展する将来領域です。

社員マスタ、店舗、部署、役職、職種、権限はCore DBが正本です。
NOV Talent側では独自マスタを作らず、必要な場合はCore DBを参照します。

現時点ではDB変更、社員マスタ更新、HR秘匿情報保存は行いません。

## MVPで最初に作るべきもの

### 1. 入社手続きキュー

最優先です。

理由:

- リクルート管理から自然につながる
- 内定後フォローの次工程として総務人事部が使いやすい
- 社員マスタに反映する前の確認待ちを可視化できる
- いきなり `employees` を更新しない安全な段階を作れる

扱う候補:

- 対象学生
- 入社予定日
- 配属予定店舗
- 希望店舗
- 必要書類
- 制服・備品準備
- 連絡完了
- 社員マスタ反映待ち
- 承認者

Coreレビュー必須:

- `talent_onboarding_tasks` などの新規DDL
- `talent_students.id` と `employees.id` の接続方針
- `stores.id` 参照
- 承認フロー
- 通知

## MVPで次に作るもの

### 2. 退職手続きチェック

理由:

- 総務人事部の抜け漏れ防止効果が大きい
- 貸与物、アカウント、最終出勤日、社会保険などのチェックが明確
- 社員マスタ更新前の確認待ちキューにしやすい

注意:

- 退職理由などセンシティブな情報はHR Module側に寄せる
- NOV Talent側では、手続き進捗と確認状態に留める

### 3. 異動手続きチェック

理由:

- 店舗間共有、配属日、異動元/異動先の確認が必要
- Core DBの店舗・異動履歴とつながる

注意:

- `employee_assignment_histories` へ直接反映する前に承認キューを置く
- 店舗名ではなく `stores.id` を参照する

### 4. 休職・復職手続きチェック

理由:

- 産休・育休、休職、復職は総務人事部の重要業務
- ただしHR秘匿情報が混ざりやすいため、MVPでは後回しが安全

注意:

- 詳細理由、診断書、家族情報、給与/保険関連はHR Moduleへ分離
- NOV Talent側では、開始日、復職予定日、確認状態程度に留める

## MVPでやらないこと

- 社員マスタの直接更新
- HR秘匿情報の保存
- マイナンバー、給与、税、保険、扶養、家族情報の管理
- 退職理由や休職理由の詳細保存
- Core DBのDDL投入
- RLS / GRANT / RPC変更
- Secret / service_role変更

## UI案

現職者管理タブ内に、まずは以下のカードを置く。

```text
現職者管理
├─ 入社手続き
│  ├─ 反映待ち
│  ├─ 書類確認
│  ├─ 配属確認
│  └─ 社員マスタ反映待ち
├─ 異動手続き
├─ 休職・復職
└─ 退職手続き
```

MVPでは、カード表示とチェックリストの設計確認まで。
保存機能はCoreレビュー後に追加する。

## 画面での見せ方

### 総務人事部の日常画面

- 今日確認する手続き
- 期限切れ
- 承認待ち
- 社員マスタ反映待ち

### 管理者画面

- 手続き種別の設定
- 承認者の確認
- Core反映待ち一覧
- 操作履歴

## 判断

最初のMVPは `入社手続きキュー` に絞るのがよいです。

理由は、リクルート管理から最も自然につながり、総務人事部の運用効果が高く、社員マスタへ直接書き込まない安全な中間ステップを作れるためです。

退職、異動、休職・復職は、入社手続きキューの型が固まってから横展開します。
